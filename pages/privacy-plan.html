PRIVACY PLAN

プライバシーを、選べる設計へ。Make privacy choices visible.

このページは、kAIgiの現在の動作と、これから議論するプライバシー機能の候補を分けて整理します。候補は確定仕様や実装済み機能ではありません。This page separates kAIgi’s current behavior from privacy features for future discussion. The candidates below are not commitments or implemented features.

議論用ドラフト · kAIgi 0.2.10Discussion draft · kAIgi 0.2.10

Current

現在の仕様Current behavior

0.2.10での境界The boundary in 0.2.10

kAIgi CLIはMac上で録音と文字起こしを扱い、CLI自体が音声や文字起こしをクラウドへアップロードすることはありません。一方、CodexやClaude CodeがCLIの出力をモデルへ渡す場合、選択したモデル提供元へメタデータや文字起こしが送られる可能性があります。アプリはCLI初回インストール時にこの点を警告しますが、現時点ではコマンドごとの確認、送信先の自動判定、名前のマスキングは行いません。The kAIgi CLI handles recordings and transcripts on this Mac and does not upload them to the cloud by itself. However, when Codex or Claude Code passes CLI output to its model, metadata or transcript text may reach the selected model provider. kAIgi warns about this before the first CLI installation. It does not yet ask before every command, detect the provider automatically, or mask names.

ローカル処理Local processing

録音、オンデバイス文字起こし、議論ログはMac上で処理します。Recording, on-device transcription, and discussion logs are processed on the Mac.

初回インストール警告First-install warning

CLIを初めてインストールする前に、モデル提供元への送信可能性を説明します。Before the first CLI installation, kAIgi explains that model providers may receive content.

AI Researchは別機能AI Research is separate

AI ResearchのOpenAI API連携はCLIとは別の設定・送信経路です。AI Research’s OpenAI API integration is a separate setting and transmission path.

Plan

今後検討する機能Features to discuss

1. 送信前の確認1. Confirm before sending

最優先候補Top candidate

クラウドまたは送信先不明のモデルへ渡る前に、何が送られるかを表示し、明示的に許可できるようにします。録音一覧のメタデータと文字起こし本文を分けて説明します。Before content reaches a cloud or unknown model, show what will be sent and let the user allow it explicitly. Explain recording metadata separately from transcript text.

2. 送信先の見える化2. Show the destination

判断の土台Decision foundation

ローカル、クラウド、判定不能を区別し、どの処理がどこへ渡る可能性があるかを設定と実行時に確認できるようにします。Distinguish local, cloud, and unknown processing so users can see where each operation may send data in Settings and at execution time.

3. 名前・固有名詞のマスキング3. Mask names and proper nouns

品質との両立Balance with quality

送信前のコピーだけを一貫した仮名へ置き換え、元の録音・文字起こしはMacに残します。マスキングによる精度低下と、元に戻せる範囲を明示します。Replace names with consistent aliases only in the copy sent out, while keeping the original recording and transcript on the Mac. Explain accuracy trade-offs and what can be restored.

4. ローカル限定モード4. Local-only mode

強い保護Strong protection

外部モデルへの送信を許可しない設定を用意し、クラウドや判定不能な経路を失敗させます。使えない機能を事前に説明します。Provide a setting that rejects cloud or unknown paths and explain in advance which features become unavailable.

5. 送信履歴の確認5. Review transmission history

後から確認Review later

いつ、どの機能が、どの種類のデータを外部モデルへ渡したかを、本文を保存せずに確認できるようにします。Let users review when a feature sent which type of data to an external model without retaining transcript content in the log.

次に決めることWhat we need to decide next

推奨順は、①送信内容と送信先を表示して明示許可、②送信前コピーのマスキング、③ローカル限定モード、④送信履歴です。まず①を決めると、ユーザーがその場で判断でき、マスキングの品質低下を無理に受け入れずに済みます。The recommended order is: (1) show the content and destination and ask explicitly, (2) mask the outgoing copy, (3) add local-only mode, and (4) add transmission history. Deciding (1) first gives users an immediate choice without forcing them to accept masking quality loss.

候補への意見や具体的な利用場面は、GitHub Issuesで受け付けます。正式な仕様に採用した機能は、実装前にこのページとプライバシーポリシーを更新します。Share use cases and feedback in GitHub Issues. Once a candidate becomes an adopted feature, this page and the Privacy Policy will be updated before implementation.
